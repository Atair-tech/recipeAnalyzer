我要开发一个运行在 Windows 上的“家庭菜谱管理软件”，主要用户是我老婆。她多年做菜积累了一个很大的 Excel 文件，里面记录了菜名、原料、做法、心得、备注等内容。随着 Excel 越来越大，且内容并不完全结构化，查找、整理、筛选、统计和复用都越来越困难。

请你作为开发代理，帮我从 0 开始搭建这个项目。目标不是做一个简单的 Excel 查看器，而是做一个“本地优先的菜谱知识库桌面软件”，后续可加入 AI 能力。

====================
一、项目目标
====================

我要的软件需要实现这些核心目标：

1. 替代 Excel 的日常查看和管理功能
- 导入现有 Excel 菜谱数据
- 浏览菜谱列表
- 查看单个菜谱详情
- 搜索和筛选
- 编辑和保存

2. 把半结构化内容逐步整理成更清晰的数据
- 把菜名、原料、步骤、心得、标签等拆分和规范化
- 保留原始 Excel 文本，避免清洗时丢失信息
- 支持后续逐步修正结构化结果

3. 做成知识库风格的软件，而不是单纯表格
- 左侧分类/标签/筛选
- 中间菜谱列表
- 右侧详情或详情页
- 支持卡片视图和列表视图
- 风格偏“知识库/笔记类应用”

4. 后续支持 AI 增强
- 自然语言搜索
- 自动标签
- 智能摘要
- 基于现有食材推荐菜谱
- 从心得中提炼注意事项/失败点/经验总结

====================
二、技术路线
====================

我目前倾向于下面这套技术栈：

- 前端：React
- 桌面壳：Tauri
- 后端：Python（FastAPI）
- 数据库：SQLite
- Excel 处理：pandas / openpyxl
- 图表：后续可用 ECharts 或 Recharts
- AI：先预留接口，后续可接云端 API 或本地 Ollama

请按这个技术路线来设计项目。

职责理解如下：

1. React 负责：
- 界面展示
- 页面布局
- 搜索/筛选交互
- 菜谱列表
- 菜谱详情
- 图表页面
- AI 助手页面

2. Tauri 负责：
- 把 React 包装成 Windows 桌面软件
- 调用本地文件系统
- 调起文件选择器
- 与本地后端通信
- 打包 Windows 应用

3. Python 后端负责：
- 解析 Excel
- 数据清洗
- 数据库读写
- 搜索逻辑
- 推荐逻辑
- AI 接口封装

====================
三、产品形态
====================

我希望软件整体偏“知识库/笔记类应用”风格，而不是传统企业后台。适合参考的产品体验：

- 左侧导航栏
- 中间列表/卡片区
- 右侧详情区 或 单独详情页
- 标签明显
- 支持搜索
- 支持像知识库一样浏览内容

页面建议包括：

1. 首页 / 总览页
- 分类入口
- 最近新增菜谱
- 收藏菜谱
- 最近做过
- 一些简单统计

2. 菜谱库页面
- 左侧：筛选器（食材、标签、菜系、耗时、难度、厨具等）
- 中间：菜谱卡片或列表
- 右侧或详情页：菜谱详情

3. 菜谱详情页
- 基本信息
- 原料
- 做法步骤
- 心得/备注
- 标签
- 图片（后续）
- AI 总结（后续）

4. 数据分析页
- 高频食材
- 菜系分布
- 常做菜
- 标签统计
- 厨具使用情况

5. AI 助手页
- 自然语言检索
- 食材推荐
- 摘要整理
- 智能标签建议

====================
四、数据设计思路
====================

请不要把 Excel 直接当数据库使用。Excel 只是导入来源。真正存储请使用 SQLite。

建议围绕以下实体设计数据库：

1. recipes（菜谱）
字段建议：
- id
- name（菜名）
- alias（别名，可空）
- category（分类，如热菜/凉菜/汤/主食等）
- cuisine（菜系，可空）
- flavor（口味，可空）
- difficulty（难度，可空）
- estimated_time（预计耗时，可空）
- servings（适合人数，可空）
- tools（厨具，可空或拆表）
- steps_text（做法原文）
- notes_text（心得原文）
- source_text（原始导入文本）
- created_at
- updated_at

2. ingredients（食材字典）
字段建议：
- id
- name
- alias
- normalized_name

3. recipe_ingredients（菜谱和食材关联）
字段建议：
- id
- recipe_id
- ingredient_id
- amount
- unit
- remark

4. tags（标签）
字段建议：
- id
- name
- type（可空）

5. recipe_tags（菜谱和标签关联）
字段建议：
- id
- recipe_id
- tag_id

6. import_batches（导入批次）
字段建议：
- id
- file_name
- imported_at
- raw_meta

7. raw_import_rows（原始导入行）
字段建议：
- id
- batch_id
- row_index
- raw_json
- parse_status
- parse_result_json

核心要求：
- 一定保留原始导入内容
- 后续解析失败时可以回溯
- 结构化结果和原始文本并存

====================
五、核心功能优先级（MVP）
====================

请先做 MVP，不要一开始做太多复杂 AI。

第一阶段优先实现：

1. 项目基础脚手架
- Tauri + React 前端工程
- Python FastAPI 后端工程
- 前后端联调
- SQLite 初始化

2. Excel 导入
- 从本地选择 Excel 文件
- 后端读取 Excel
- 显示导入预览
- 保存到数据库
- 保留原始数据

3. 菜谱列表浏览
- 列表视图
- 卡片视图（可后续）
- 关键词搜索
- 基础筛选

4. 菜谱详情查看
- 菜名
- 原料
- 做法
- 心得
- 备注
- 标签

5. 基础编辑能力
- 修改菜名、标签、分类等
- 修改做法和心得
- 保存回数据库

6. 基础筛选
- 按食材
- 按标签
- 按分类
- 按耗时（后续可补）
- 按难度（后续可补）

====================
六、后续功能（第二阶段）
====================

后续可继续做：

1. 数据整理
- 原料自动拆分
- 别名归一化（如番茄/西红柿）
- 重复菜谱检测
- 标签建议

2. 可视化
- 高频食材统计
- 菜系/分类分布
- 最近新增趋势
- 标签排行

3. 体验优化
- 收藏
- 最近做过
- 做过次数
- 推荐菜单
- 买菜清单

====================
七、AI 功能规划（先预留，不必首版完成）
====================

后续希望接入 AI，建议架构上预留 service 层和 API 接口。

未来可能实现：

1. 自然语言搜索
例子：
- 想吃清淡一点的
- 30 分钟内能做好的
- 用鸡肉但不辣的菜
- 以前做过哪些适合请客的菜

2. 自动标签
根据菜谱正文生成：
- 快手
- 下饭
- 宴客
- 新手友好
- 清淡
- 重口味
等标签

3. 心得整理
从原始心得中提取：
- 关键步骤
- 易失败点
- 替代食材
- 调味建议

4. 食材推荐
输入现有食材，推荐能做的菜谱，或差一点什么就能做

====================
八、开发要求
====================

请你作为开发代理时遵循这些原则：

1. 先做可运行的 MVP，再迭代
2. 代码结构清晰，便于后续扩展
3. 前后端解耦
4. 不要把逻辑写死在界面层
5. 优先保证本地数据可靠存储
6. 导入和清洗过程要可回溯
7. 尽量使用易维护方案，不要过度设计
8. 对每一步给出明确的文件结构、代码实现和运行说明

====================
九、我希望你现在先做什么
====================

请先按以下顺序开始：

第一步：
- 设计整个项目目录结构
- 明确前端、Tauri、Python 后端、数据库各自放在哪里
- 给出初始化方案

第二步：
- 搭建一个最小可运行版本
- React 前端能启动
- Tauri 能运行桌面窗口
- Python FastAPI 能启动
- 前端能调用后端一个测试接口
- SQLite 能初始化

第三步：
- 实现 Excel 导入的第一版
- 允许选择本地 Excel 文件
- 后端读取文件
- 把读取结果展示到前端预览表格
- 再保存到数据库

第四步：
- 实现菜谱列表页和详情页基础版

请你直接开始产出项目骨架、目录结构、关键代码和执行步骤，不要只给空泛建议。
