# Recipe Analyzer 用户手册

## 1. 这是什么

`Recipe Analyzer` 是一个本地运行的菜谱库应用，用来把你维护在 Excel 工作簿里的菜谱，整理成一个可浏览、可检索、可筛选、可分析的本地数据库。

当前版本的核心前提：

- Excel 仍然是主数据源
- Web 端目前只读，不直接编辑
- 系统按差异同步 Excel，不是简单追加导入

## 2. 当前支持的 Excel 结构

当前导入器是按真实工作簿设计的，支持这些工作表类型：

- 成对工作表
  - 例如 `牛肉 / 牛肉做法`
  - 前者负责目录和分类
  - 后者负责食材、调料、做法
- 独立工作表
  - `甜点配方专区`
- 待办工作表
  - `再挑战及待记录`

系统会把这些内容拆成结构化字段：

- 正式菜谱 / 待办项
- 专题库
- 分组
- 菜名
- 菜系 / 亚菜系
- 最后记录日期
- 来源或修订备注
- BMD / CC 标记
- 食材
- 调料
- 做法及要点

## 3. 怎么打开

如果开发环境已经启动，直接在浏览器打开：

- 前端页面：`http://127.0.0.1:5173`
- 后端接口：`http://127.0.0.1:8000`

## 4. 页面说明

### 4.1 总览

总览页会显示：

- 正式菜谱数量
- 待办项数量
- 专题库数量
- 最近更新的一条记录

### 4.2 菜谱库

这是日常使用最频繁的页面。

支持：

- 关键词搜索
- 按类型筛选
  - 正式菜谱
  - 待挑战
  - 待记录
- 按专题库筛选
- 按分组筛选
- 按菜系筛选
- 按食材筛选
- 仅看 `BMD`
- 仅看 `CC`

列表卡片会显示：

- 菜名
- 类型
- 专题库 / 分组 / 菜系
- 亚菜系
- BMD / CC
- 最后记录日期或来源备注

### 4.3 详情页

详情页是只读的。

对于正式菜谱，会显示：

- 专题库
- 分组
- 菜系 / 亚菜系
- 最后记录日期
- 来源或修订备注
- 食材
- 调料
- 做法及要点
- 结构化食材
- 系统备注
- 原始导入文本

对于待办项，会显示：

- 状态
  - `待挑战` 或 `待记录`
- 来源工作表
- 原始导入文本

### 4.4 同步

同步页用于把 Excel 更新到 SQLite。

流程：

1. 选择工作簿文件
2. 点击 `预览结构`
3. 检查解析结果
4. 点击 `同步到 SQLite`

预览阶段会显示：

- 总记录数
- 正式菜谱数
- 待办项数
- 已配对条目数
- 仅索引页条目数
- 仅做法页条目数
- 工作表列表
- 专题库分布
- 预览记录表

### 4.5 导入历史

导入历史会记录每次同步批次。

可以查看：

- 批次编号
- 导入时间
- 导入文件名
- 本批次的记录数
- 正式菜谱 / 待办项数量
- 仅索引页 / 仅做法页数量
- 本批次的工作表列表
- 本批次的预览记录

### 4.6 分析

分析页会从当前数据库生成数据审视结果。

包括：

- 专题库分布
- 分组分布
- 菜系分布
- 待办分布
- 高频食材
- 缺少做法的专题库
- 最近同步批次

### 4.7 AI

当前 AI 页是基础版。

支持：

- 自然语言检索
- 自动标签建议

自然语言检索会结合：

- 专题库
- 分组
- 菜系
- 食材
- 全文内容

标签建议目前是：

- 规则建议
- 相似记录建议

## 5. 差异同步规则

系统不是每次“清空重导”，也不是每次“无脑追加”。

每次同步时，会把 Excel 与当前数据库对比，然后得到四类结果：

- `新增`
- `更新`
- `删除`
- `未变化`

这意味着：

- 同一份工作簿重复导入，不会无限重复
- 没变的条目不会重复写入
- 被 Excel 删除的条目，也会从数据库里删除

## 6. 当前为什么不能在页面里编辑

当前版本明确采用：

- Excel 是主数据源
- Web 端以浏览和检索为主

如果同时允许 Web 端直接编辑，而这些编辑又不回写 Excel，下次重新同步就会被 Excel 覆盖。

因此当前正确做法是：

1. 修改 Excel
2. 保存 Excel
3. 回到系统重新同步

## 7. 使用建议

### 7.1 真实数据维护建议

建议把这套系统理解为：

- Excel 负责维护原始内容
- SQLite 负责浏览、检索、分析和后续 AI 增量处理

### 7.2 如果以后要做增量 AI

当前的差异同步机制已经适合后续 AI 增量处理。

未来可以只对这些记录做 AI：

- 新增
- 更新

而不必每次对全部菜谱重新处理一遍。

### 7.3 仍然建议补“稳定编号”

目前系统主要依赖名称和工作簿结构做匹配。

如果将来你在 Excel 里频繁改菜名，建议增加稳定编号列，后续可以进一步把同步做得更稳。

## 8. 常见问题

### 8.1 为什么有“仅索引页”或“仅做法页”

因为真实工作簿里有些条目只出现在目录页，或者只出现在做法页。

系统不会直接丢掉这些内容，而是照样入库，并在分析与详情中保留它们。

### 8.2 为什么有些记录是“待挑战”或“待记录”

这些来自 `再挑战及待记录` 工作表，不属于正式菜谱，但会作为待办项保留在库里。

### 8.3 如果我重复同步同一份 Excel 会怎样

如果文件内容没有变化，理论上结果会全部落在 `未变化`。

### 8.4 数据存在哪里

当前数据库文件默认在：

`data/recipe_analyzer.db`

原始工作簿在：

`data/recipes.xlsx`

## 9. 最简操作路径

如果只记最短流程，按下面做即可：

1. 修改 `data/recipes.xlsx`
2. 打开 `同步`
3. 先 `预览结构`
4. 再 `同步到 SQLite`
5. 去 `菜谱库` 浏览和筛选

---

这份手册不要求每次代码变更后自动更新，按需要维护即可。
