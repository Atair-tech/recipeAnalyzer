# AI 精校优化方案

## 1. 背景

当前库中，`qwen3:0.6b` 已完成 `435` 条 recipe 的 AI 精校切分。已落库状态：

- `processed_count = 435`
- `refined_count = 426`
- `error_count = 9`

对这批结果做抽样和数据库统计后，结论是：

- 这批结果已经能处理一部分简单、规整的原料文本。
- 但在复杂原文、替代项、括号说明、多候选项、品牌和备注混杂的情况下，错误率已经足够明显，不适合直接沿用当前一段式流程继续全量跑。

## 2. 当前发现的主要问题

### 2.1 多候选项未拆开，直接作为单个食材名落库

典型例子：

- `粿条/粉丝`
- `牛腱子/牛肋条`
- `鸡腿/整鸡`
- `鸡胸肉or鸡腿肉or整鸡`
- `海带/裙带菜`

问题：

- 这类内容不是单一食材实体。
- 直接落为 `ingredients.name` 会污染食材词典和筛选下拉。

正确方向：

- 拆成多个候选食材，非主选项放 `remark=可选`
- 或保留主项，其余进 `remark`

### 2.2 备注、建议、说明句漏进了食材名

典型例子：

- `更好的选择是加猪蹄`
- `土豆等煲类常用食材`
- `南瓜土豆红薯茄子豆角彩椒`
- `云南番茄腐乳胡辣鱼`
- `姜食堂烤牛肉意面`

问题：

- 这些不是食材本体，而是建议句、成菜名、说明短语、并列描述。

正确方向：

- `ingredients.name` 只保留可单独采购或识别的食材实体
- 其他信息丢弃或转入 `remark`

### 2.3 数字、比例、品牌残片进入食材

典型例子：

- `%`
- `1`
- `六必居雪菜半包40g`
- `暴肌独角兽橄榄油葱油汁`

问题：

- 当前流程没有稳定剥离品牌、规格、比例残片。

正确方向：

- 纯噪声直接过滤
- 品牌前缀剥离后再归一到食材实体

### 2.4 模型改写原文时反向污染结构化结果

典型例子：

- `薄荷记（200g）`
- `鸡胸肉100%（也可以用鸡腿肉；%）`
- `牛肚（和牛蹄筋，最好加，更好的选择是加猪蹄）`

问题：

- 现在模型同时负责：
  - 改写 `ingredients_text`
  - 生成 `ingredients[]`
- 一旦文本改写变脏，后续结构化结果会继续被带偏。

正确方向：

- 不要让模型自由重写 `ingredients_text`
- 优先让模型只输出结构化食材，再由程序反向生成展示文本

### 2.5 任务定义过宽，小模型负担过重

当前一次调用同时要求模型处理：

- `ingredients_text`
- `seasonings_text`
- `steps_text`
- `notes_text`
- `ingredients[]`

问题：

- `qwen3:0.6b` 对这种多字段、多目标、强约束任务不稳
- 容易在字段之间串味，造成错误扩散

## 3. 数据判断

对 `qwen3:0.6b` 已精校样本统计：

- 精校过的 recipe：`435`
- 结构化食材关联：`1572`
- 唯一食材词：`552`
- 至少存在明显坏食材的 recipe：`21`

说明：

- `21` 条只是“明显坏掉”的下限，不包含大量“看起来像食材，但仍不够干净”的软错误。
- 这已经足以说明当前流程不能直接用于最终全量精校。

## 4. 优化方向总览

结论不是“只改 prompt”，而是同时改：

1. `prompt`
2. `后处理`
3. `任务流程`

优先级：

1. 先改流程
2. 再补强后处理
3. 最后再调 prompt

## 5. 推荐的新流程

### 5.1 第一阶段：只精校结构化食材

模型只输出：

- `ingredients[]`

每项字段固定为：

- `name`
- `amount`
- `unit`
- `remark`

不再同时要求模型改写：

- `ingredients_text`
- `seasonings_text`
- `steps_text`
- `notes_text`

原因：

- 食材结构化是当前最关键、最脆弱的部分
- 先把最关键链路做稳

### 5.2 第二阶段：由程序反向生成展示文本

如果页面还需要新的 `ingredients_text`，不让模型自由改写。

改为：

- 根据精校后的 `ingredients[]`
- 由程序统一拼回展示文本

收益：

- 避免模型把短语、备注、建议句重新写回原文字段
- 文本格式可控

### 5.3 第三阶段：加强后处理

模型输出后，再做程序侧兜底。

至少增加以下规则：

- 含 `/`、`or`、`或` 的，拆分或改成 `remark=可选`
- 纯数字、纯 `%`、纯符号，直接丢弃
- 包含 `推荐/建议/如果/或者/需要/更好吃/常用食材/原版` 的，默认不允许进入 `name`
- 品牌前缀剥离
- 成菜名、套餐名、做法名黑名单过滤

### 5.4 第四阶段：单独精校调料 / 备注

后续如果继续做：

- `seasonings_text`
- `notes_text`

也应单独调用模型，不再与食材结构化共用一个 prompt。

## 6. Prompt 优化建议

即使采用新流程，Prompt 也要收紧。

推荐目标：

- 明确要求 `name` 只能是食材实体
- 明确给出“什么不能算食材”
- 明确规定多候选项、可选项、替代项的处理方式

### 6.1 Prompt 要点

需要明确写进 prompt 的规则：

1. `name 必须是可单独采购的原料实体`
2. `绝不允许输出成菜名称`
3. `绝不允许输出建议句、说明句、口味描述、品牌宣传`
4. `A/B、A or B、A（也可用B）必须拆开`
5. `可加A / 加A更好吃 / 可放B / 也可用C`
   - 处理为：
     - `name = A/B/C`
     - `remark = 可选`
6. `不能把 %、1、半包40g 这类残片作为 name`
7. `只输出 JSON，不输出解释`

### 6.2 推荐的新 Prompt 草案

```text
你负责把菜谱中的原料文本整理为结构化食材数组。

只返回 JSON，不要输出解释，不要输出 Markdown。

目标：
- 只保留“食材本体”
- 不要保留短语、句子、建议、做法说明、品牌宣传、比例说明、符号残片

严格规则：
1. 每个 ingredient 必须是可单独采购或明确识别的原料实体。
2. name 只能是食材本体，例如：番茄、鸡蛋、豆腐、牛肋条。
3. 以下内容不能作为 name：
   - 成菜名
   - 套餐名
   - 建议句
   - 口味描述
   - 比例说明
   - 品牌宣传
   - 标点或符号残片
4. 对于 “A/B”、“A or B”、“A（也可用B）”：
   - 拆成多个 ingredient
   - 非主选项写 remark=可选
5. 对于 “可加A”、“加A更好吃”、“可放B”、“也可用C”：
   - 输出 name=A/B/C
   - remark=可选
6. 如果某段文本不是食材实体，直接丢弃。
7. 不要发明原文中不存在的信息。

输出格式：
{
  "ingredients": [
    {
      "name": "",
      "amount": "",
      "unit": "",
      "remark": ""
    }
  ]
}
```

## 7. 后处理规则建议

### 7.1 必加规则

- 丢弃纯噪声：
  - `%`
  - `1`
  - `+`
  - `*`
  - `/`
  - `=`

- 拦截说明词：
  - `推荐`
  - `建议`
  - `如果`
  - `或者`
  - `需要`
  - `更好吃`
  - `常用食材`
  - `原版`

- 处理候选项：
  - `鸡腿/鸡翅`
  - `海带/裙带菜`
  - `牛腱子/牛肋条`

### 7.2 可选增强

- 品牌词剥离词典
- 食材别名归一词典
- 成菜名黑名单
- 采购备注识别

## 8. 模型选择建议

### 8.1 对 `qwen3:0.6b` 的判断

适合：

- 流程验证
- Prompt 调试
- 小样本对照试验

不适合：

- 当前这种全字段一段式精校
- 最终全量正式跑库

### 8.2 对 `qwen3:4b` 的建议

建议把 `qwen3:4b` 作为正式精校主模型，但前提是：

1. 先拆流程
2. 先加后处理
3. 再做小样本 A/B 测试

不要只是把当前 prompt 直接换成 4b 再全量跑。

## 9. 推荐实施步骤

### 第一步

把精校功能改成：

- 模型只输出 `ingredients[]`

### 第二步

增加强后处理：

- 候选项拆分
- 建议句过滤
- 品牌和比例残片过滤

### 第三步

做 `50-100` 条样本对照测试：

- 旧流程 + `qwen3:0.6b`
- 新流程 + `qwen3:4b`

### 第四步

抽样人工审查结果：

- 食材实体准确率
- 脏项残留率
- 候选项处理是否合理

### 第五步

确认质量后再全量跑库。

## 10. A/B 测试建议

建议抽取三类 recipe：

1. 简单型
   - 原料行规整、几乎没有备注
2. 中等复杂型
   - 有括号、有替代项、有少量建议语
3. 高复杂型
   - 多括号、多品牌、多候选项、来源文本混杂

对每类各抽：

- `15-30` 条

观察指标：

- 是否出现成菜名入库
- 是否出现句子短语入库
- 是否出现 `% / 1 / 符号残片`
- 是否正确处理候选项和可选项
- 是否把建议句正确转成 `remark=可选`

## 11. 最终建议

当前最值得做的不是继续让 `qwen3:0.6b` 直接跑剩余全库，而是：

1. 先重构精校任务为“只精校结构化食材”
2. 增加强后处理
3. 再用 `qwen3:4b` 做小样本验证
4. 通过后再全量跑

否则只是把当前问题继续扩大到全库。
